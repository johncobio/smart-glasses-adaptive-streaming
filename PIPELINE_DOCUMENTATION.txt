================================================================================
SYSTEM BASELINE — MENTRAOS OPEN-SOURCE INFRASTRUCTURE
================================================================================

All three pipelines are built on top of the Mentra Live glasses hardware and
the open-source MentraOS ecosystem (github.com/Mentra-Community/MentraOS,
MIT License).

HARDWARE
  Mentra Live glasses
    - Camera: front-facing, fixed FOV
    - Connectivity: Bluetooth to Android companion phone (no direct WiFi/LTE)
    - Firmware: closed (not modifiable)
    - The glasses have no independent internet connection. All data — video,
      commands, and sensor events — flow through the companion phone.

  Android companion phone
    - Runs the MentraOS app (React Native, TypeScript)
    - Bridges glasses↔Bluetooth to the cloud over WiFi or LTE
    - Responsible for RTMP stream encoding and transmission

MENTRACLOUD (api.mentraglass.com)
  The original MentraOS architecture routes all traffic through MentraCloud:
    Glasses <--BT--> Phone App --> MentraCloud --> RTMP destination

  MentraCloud's relay pipeline (cloud/rtmp_relay/stream-manager.sh) includes
  two intentional delay mechanisms that are hardcoded and cannot be disabled:
    1. sleep 2       — 2000 ms "stream stabilization" wait before forwarding
    2. ffmpeg -analyzeduration 3M — 500–1000 ms probe before processing

  These produce a consistent ~3,208 ms additional latency on top of the
  firmware-level video prebuffer. MentraCloud also adds a routing hop
  (MentraCloud → MediaMTX) that averages ~947 ms measured in Pipeline 2.

FIRMWARE PREBUFFER
  The glasses firmware buffers video locally before pushing RTMP regardless
  of which server receives the stream. This prebuffer is documented by Mentra
  at approximately 3 seconds and cannot be reduced without firmware access.
  It is the irreducible floor for stream startup latency in all pipelines.

PIPELINE 1 — TWITCH / NGINX ADAPTIVE STREAMING
================================================================================

Dates:        February 5 – March 4, 2026
Sessions:     21 logged sessions
Purpose:      Demonstrate real-time adaptive video quality based on head
              motion, streaming to Twitch as a public live broadcast.

────────────────────────────────────────────────────────────────────────────────
1.2  KEY COMPONENTS
────────────────────────────────────────────────────────────────────────────────

  mentra-app/src/index.ts
    - Registers with MentraCloud as an SDK session handler
    - Receives head position events (UP / DOWN) from the glasses via SDK
    - Maps events to synthetic angular velocity:
        head DOWN → 200 dps (simulating fast motion)
        head UP   → 0 dps   (simulating stillness)
    - Forwards motion data to Python backend via WebSocket message:
        { type: "MOTION", angularVelocity: <dps>, timestamp: <ms> }

  mentra-app/src/StreamManager.ts
    - Controls RTMP stream start/stop via Mentra SDK
    - Tracks stream status events (streaming / stopped / error)
    - Receives SET_QUALITY commands from Python backend and forwards to SDK
      Note: quality adaptation was tested but the pipeline ultimately ran
      on standard Mentra Live streaming with the SDK managing quality natively

  mentra-app/src/BackendClient.ts
    - WebSocket client connecting the Node.js app to the Python backend
    - Runs on localhost, port 8765
    - Sends MOTION messages; receives SET_QUALITY commands

  backend/src/adaptation_controller.py
    - WebSocket server (port 8765) receiving motion data from Node.js
    - Implements a state machine with three states:
        STABLE       — head still, stream at highest quality
        FAST         — fast motion detected, stream at lowest quality
        TRANSITIONING — motion stopped; hysteresis prevents rapid toggling
    - Configurable thresholds (backend/src/config.py):
        FAST_MOTION_THRESHOLD_DPS = 120   (above → FAST state)
        SETTLE_TIME_MS = 2000             (wait before upgrading quality)
    - Sends SET_QUALITY commands to Node.js when state transitions occur
    - Measures round-trip command latency: time from SET_QUALITY sent
      to acknowledgement received back through MentraCloud

  backend/src/metrics_logger.py
    - Writes per-session CSV files to backend/logs/
    - File naming: YYYYMMDD_HHMMSS_performance.csv
    - Columns (11 total): timestamp_iso, timestamp_ms, quality_level,
      motion_state, angular_velocity_dps, client_last_cmd_latency_ms,
      client_rtt_ms, client_avg_rtt_ms, server_imu_proc_avg_ms,
      session_id, notes

  backend/src/config.py
    - Centralizes all tunable constants
    - Motion detection thresholds (FAST_MOTION_THRESHOLD_DPS = 120)

────────────────────────────────────────────────────────────────────────────────
1.3  NETWORKING
────────────────────────────────────────────────────────────────────────────────

  Challenge: The Linux desktop is behind a home router with no port forwarding.
  MentraCloud requires a publicly reachable URL to deliver SDK events to
  the Node.js app.

  Solution: ngrok tunnel
    ngrok http 3000
    → https://<random>.ngrok-free.app forwarded to localhost:3000

  Limitation: The ngrok URL changes on every restart. It must be manually
  updated at console.mentra.glass before each session. The free tier also
  imposes rate limits on HTTP requests.

  RTMP destination: rtmp://live.twitch.tv/live/<stream-key>
  The glasses push RTMP directly through MentraCloud's relay to Twitch.

────────────────────────────────────────────────────────────────────────────────
1.5  LIMITATIONS IDENTIFIED
────────────────────────────────────────────────────────────────────────────────

  1. No server-side stream verification
     Quality commands are sent but there is no way to confirm whether the
     glasses executed them or what the resulting bitrate was. The research
     question (does adaptive quality help?) cannot be answered without this.

  2. Twitch is a public stream
     Cannot control who views the stream. Privacy concern for a research
     context where the camera wearer may not want public exposure.

  3. ~939 ms command latency is too slow for responsive adaptation
     Head motion events are detected within milliseconds, but the quality
     change takes nearly 1 second to execute end-to-end via MentraCloud.
     By the time the command reaches the glasses, the motion event that
     triggered it has already passed.

  4. ngrok URL changes every boot
     Operational friction — requires manual update at the developer console
     before every session. The random URL also prevents scripted startup.

  5. MentraCloud relay still present
     The ~3,208 ms stream startup delay and relay overhead cannot be measured
     or eliminated in this pipeline.

2.5  LIMITATIONS IDENTIFIED
────────────────────────────────────────────────────────────────────────────────

  1. MentraCloud relay is still present
     The ~3,208 ms firmware prebuffer plus ~947 ms MentraCloud relay overhead
     produce a minimum ingest latency of ~4,000 ms in typical conditions.
     This is hardcoded in MentraCloud's relay script and cannot be reduced.

  2. Bluetooth hop is unobservable
     There is no way to measure the latency of the glasses→Bluetooth→phone
     link. The SDK treats this as an internal detail and does not expose
     per-frame Bluetooth timing.

  3. ngrok still required
     The MentraOS companion app must be able to receive events from
     MentraCloud, which requires a public URL. ngrok is still needed and
     its URL still changes on reboot.

  4. HLS viewer latency is high
     HLS requires the player to buffer multiple segments before playback
     begins. Measured viewer latency: approximately 13 seconds end-to-end
     (stream captured → viewed in VLC / ffplay via HLS). This is a
     protocol-level constraint of HLS, not a pipeline bug.

  5. Startup delay variance is not fully explained
     The range 1,386–3,208 ms across identical hardware/network conditions
     suggests the prebuffer warm/cold state significantly affects results.
     The mechanism for warming is not documented by Mentra.

3.3  KEY COMPONENT — glasses_ws_server/server.py
────────────────────────────────────────────────────────────────────────────────

  ~350 lines of Python. Entirely new file — does not build on Pipeline 1/2.
  Uses aiohttp (not the websockets library) because the phone app sends HTTP
  POST requests before opening the WebSocket, and the websockets library's
  process_request hook only handles GET — it silently rejects POSTs at the
  HTTP parsing layer.

  Server binding:
    HOST: 0.0.0.0  (all interfaces — WiFi and Tailscale)
    PORT: 8766

  REST ENDPOINTS (phone app calls these on startup before WebSocket):
    POST /auth/exchange-token
      → {"coreToken":"bypass-token"}
      The app sends a Supabase JWT and expects a MentraCloud core token back.
      We return a dummy token to unblock the auth flow.

    POST /api/client/min-version
      → {"success":true,"data":{"required":"0.0.0","recommended":"0.0.0"}}

    POST /api/client/apps
      → {"success":true,"data":[]}
      Returns empty app list (no third-party apps installed).

    POST /api/auth/generate-webview-token
    POST /api/auth/generate-webview-signed-user-token
      → {"success":true,"data":{"token":"bypass-token"}}

    POST /api/client/device/state
    POST /api/client/audio/configure
    POST /api/client/goodbye
      → {"success":true}

    GET  /api/client/user/settings
      → {"success":true,"data":{}}

    GET|POST /{anything else}
      → {"success":true,"data":null}
      Catch-all for endpoints we don't need to handle specifically.

  WEBSOCKET HANDLER (GET /glasses-ws):
    On connection:
      1. Reset session state
      2. Record phone_connected timestamp
      3. Send connection_ack immediately
      4. Start 3-second fallback timer (_start_stream_fallback)
      5. Start keep_alive_loop (sends keep_stream_alive every 10 s)

    Messages received:
      ping
        → sends {"type":"pong"} immediately (keepalive)

      glasses_battery_update
        → records battery level and charging state
        → if stream_status == "idle": resets glasses_connected flag and
          calls _send_start_stream() — this handles the case where the
          fallback fired before glasses were ready (battery update is the
          definitive signal that glasses are connected and ready)

      glasses_connection_state  (status: CONNECTED / DISCONNECTED)
        → if CONNECTED: calls _send_start_stream() immediately
        → Note: this message is defined in MentraOS SocketComms.ts but
          sendGlassesConnectionState() is never called in practice. The
          battery update path above is the reliable trigger.

      rtmp_stream_status / stream_status
        → records stream state transitions (idle/initializing/streaming/
          active/error/reconnecting/reconnected)
        → on first transition to streaming/active/reconnected:
          records stream_active_ms timestamp for startup delay calculation
        → appends row to *_bypass_stream_events.csv

      head_position
        → records current head position (UP / DOWN / etc.)

      VAD (voice activity detection)
        → silently ignored

    _send_start_stream(ws, reason):
      Guards against double-firing (glasses_connected flag).
      Records stream_request_sent_ms timestamp.
      Logs glasses_connected latency event.
      Sends: {"type":"start_stream","streamUrl":"rtmp://136.116.71.233:1935/live/stream"}
      Logs start_stream_sent latency event.

    _start_stream_fallback(ws):
      Waits 3 seconds after phone connection.
      If glasses_connected is still False, calls _send_start_stream().
      This handles the case where the glasses were already connected when
      the phone app opened (no connection event fires in this scenario).

  MEDIAMTX POLLING (runs every 2 seconds in background):
    Calls GET http://136.116.71.233:9997/v3/paths/list
    Extracts bytesReceived, bytesSent, ready, readers
    Computes encoded_bitrate_bps and delivered_bitrate_bps from byte deltas
    On first poll returning ready=true: records mediamtx_ready_ms timestamp

  CSV LOGGING — three files per session, prefix: YYYYMMDD_HHMMSS

    *_bypass_latency.csv
      Fields: timestamp_iso, event, elapsed_from_start_ms,
              elapsed_from_prev_ms, notes
      Events logged: phone_connected, connection_ack_sent, glasses_connected,
                     start_stream_sent, stream_active, mediamtx_ready,
                     phone_disconnected
      Purpose: precise per-event latency timeline for the session

    *_bypass_performance.csv
      Fields: timestamp_iso, timestamp_ms, glasses_connected, stream_status,
              glasses_battery_level, glasses_battery_charging, head_position,
              mediamtx_ready, mediamtx_encoded_bitrate_bps,
              mediamtx_delivered_bitrate_bps, mediamtx_bytes_received,
              mediamtx_bytes_sent, mediamtx_active_readers,
              startup_delay_ms, mediamtx_ingest_latency_ms,
              server_processing_ms
      Logged every 2 seconds throughout the session.
      server_processing_ms: time taken by the Python server to execute the
      MediaMTX API poll and compute derived metrics (typically 1–5 ms).

    *_bypass_stream_events.csv
      Fields: timestamp_iso, timestamp_ms, status, stream_id, rtmp_url, error
      One row per stream status change event from the phone app.

────────────────────────────────────────────────────────────────────────────────
3.4  STARTUP TRIGGER LOGIC (detail)
────────────────────────────────────────────────────────────────────────────────

  The startup sequence has two cases:

  Case A — Glasses connect AFTER phone app opens (typical):
    t=0       Phone connects → connection_ack sent → 3s fallback started
    t~1–2s    glasses_battery_update arrives → start_stream fired immediately
    Result:   start_stream fires at glasses connection time (~1–2s from start)

  Case B — Glasses already connected when phone app opens:
    t=0       Phone connects → connection_ack sent → 3s fallback started
    t=3s      Fallback fires → start_stream sent (glasses ready but no event)
    Result:   start_stream fires at 3s regardless of how long glasses have
              been connected.

  Case C — Battery arrives before stream is confirmed active (bug, fixed):
    Fallback fires at 3s → start_stream sent → stream still idle →
    battery_update arrives → detects stream_status == "idle" → resets
    glasses_connected flag → retries start_stream.
    This handles cases where the fallback fired too early (before glasses
    had fully initialized Bluetooth).

────────────────────────────────────────────────────────────────────────────────
3.5  RELAY OVERHEAD — WHY IT DOES NOT EXIST IN PIPELINE 3
────────────────────────────────────────────────────────────────────────────────

  In Pipeline 2, relay overhead = ingest_latency − startup_delay > 0
  because MentraCloud adds delay between the SDK reporting "active" and
  the stream actually arriving at MediaMTX.

  In Pipeline 3, ingest_latency < startup_delay:
    The stream arrives at MediaMTX (1,566 ms after start_stream) BEFORE
    the phone's SDK reports the stream as "active" (2,820 ms).

  This is structurally correct: without a relay, the RTMP packets travel
  directly from the phone to MediaMTX over TCP, so MediaMTX sees them as
  soon as the glasses begin pushing. The phone-side SDK "active" status
  update lags behind the actual stream arrival because it is a higher-level
  application state callback, not a raw network event.

  Timeline comparison:
    P2:  start_stream → [2,659ms] → SDK "active" → [947ms] → MediaMTX ready
    P3:  start_stream → [1,566ms] → MediaMTX ready → [1,254ms] → SDK "active"

  The relay overhead metric (947 ms in P2) is completely eliminated in P3.
  MediaMTX receives the stream 1,254 ms earlier than the SDK even acknowledges
  it is streaming.

────────────────────────────────────────────────────────────────────────────────
3.7  LIMITATIONS
────────────────────────────────────────────────────────────────────────────────

  1. Firmware prebuffer is irreducible
     The ~2,820 ms startup delay is firmware-level: the glasses buffer video
     locally before pushing RTMP regardless of relay. Cannot be reduced
     without modifying glasses firmware (closed source).

  2. Small sample size
     Only 2 sessions have complete stream data as of the first operational
     day. More sessions are needed for statistical confidence.

  3. Ongoing display latency not automatically logged
     The ~400 ms end-to-end latency was measured manually by stopwatch.
     Logging it requires a synchronized video reference (e.g., an NTP-
     synced timestamp rendered on a monitor captured by the glasses camera,
     compared to the viewer output frame).

  4. No quality adaptation
     Pipeline 3 removes adaptive quality control (present in Pipeline 1).
     The bypass currently sends a single fixed start_stream command with no
     dynamic quality adjustment.

  5. Closed-source glasses firmware
     The Bluetooth protocol between glasses and phone is not documented.
     The Bluetooth hop latency is unobservable from our server.

================================================================================
KEY FINDINGS FOR RESEARCH PAPER
================================================================================

1. MENTRACLOUD RELAY IS THE DOMINANT LATENCY SOURCE
   Comparing P2 ingest (3,606 ms avg) to P3 ingest (1,566 ms clean):
   the 2,040 ms reduction is almost entirely from eliminating the relay.
   The relay contributes ~947 ms measured overhead on top of the prebuffer.
   P3's 61% ingest latency reduction confirms the relay, not the firmware,
   was the bottleneck.

2. THE FIRMWARE PREBUFFER IS THE IRREDUCIBLE FLOOR
   The ~2,820 ms startup delay in P3's clean session represents the glasses
   encoding and buffering video locally before any RTMP push occurs. This
   matches Mentra's documented ~3 s prebuffer. No pipeline change can
   reduce this without firmware access.

3. MENTRACLOUD RELAY INVERTS THE LATENCY RELATIONSHIP
   In P2: SDK "active" arrives before MediaMTX ready (+947 ms gap).
   In P3: MediaMTX ready arrives before SDK "active" (−1,254 ms gap).
   Removing the relay inverts the timing — the server sees the stream
   before the phone-side application layer does.

4. COMMAND LATENCY IS UNACCEPTABLY HIGH FOR REAL-TIME ADAPTATION (P1)
   The 939 ms average command round-trip via MentraCloud means quality
   commands arrive nearly 1 second after the motion that triggered them.
   The 24 ms result from a same-network session shows the hardware is
   capable of near-real-time response — the latency is MentraCloud's.

5. PROTOCOL CHOICE DOMINATES VIEWER LATENCY
   Switching from HLS (13 s) to direct RTMP playback (2–3 s) saves ~10 s
   of viewer-side latency independent of backend pipeline choice. Viewer
   latency is primarily a protocol decision, not an infrastructure decision.

6. BACKEND RTT IMPROVED 12× FROM P1 TO P2
   Removing ngrok (P1: 11–14 ms) in favor of direct localhost WebSocket
   (P2: 1.1 ms avg) reduced RTT by ~12×. This improvement confirms that
   the ngrok relay was adding measurable overhead to every performance
   metric message, not just the video path.
